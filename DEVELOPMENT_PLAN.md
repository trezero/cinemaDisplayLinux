# Cinema Display Linux — CLI Tool Development Plan

**Version:** 1.0
**Date:** 2026-03-26
**Target OS:** Ubuntu 24.04 LTS (kernel 6.8+)
**Primary Test Hardware:** Dell XPS 15 9520 + Apple LED Cinema Display 24" (A1267)

---

## 1. Goals

Build a single-file, menu-driven Python CLI tool (`cinema-display-setup`) that:

1. **Detects** Apple Cinema Displays connected via USB-C → Mini DisplayPort adapters
2. **Diagnoses** why a display isn't working and tells the user exactly what's wrong
3. **Fixes** the system configuration automatically (with sudo where required)
4. **Configures** display layout (extend, mirror, position)
5. **Controls** brightness on connected Apple displays
6. **Persists** settings across reboots
7. **Handles** known failure modes (suspend/resume, no signal, first-run setup)

---

## 2. Language & Libraries

**Language:** Python 3 (already at 3.13 on target system)

**Dependencies (all already installed on target):**
- `rich` — colored output, tables, panels, progress bars
- `click` — optional CLI argument parsing for non-interactive invocation
- `curses` — stdlib, used for the interactive menu only if needed (prefer simple numbered menus for accessibility)

**No new pip installs required.** The tool will check for its own dependencies at startup and warn gracefully if anything is missing.

**Why Python over Bash:**
- `rich` gives a polished TUI with zero extra dependencies
- Easier to unit-test individual detection/fix functions
- Cleaner error handling and state management across multi-step workflows
- More portable across Ubuntu versions

## 6. Detection Logic (Critical Path)

The tool must correctly identify whether the display is:

| State | USB detected | xrandr connected | Action |
|---|---|---|---|
| A | No | No | Guide user: check physical connection |
| B | No | Yes | Display working but no USB — guide to connect USB cable for brightness |
| C | Yes | No | Display connected via USB but no video signal — run diagnosis |
| D | Yes | Yes | Fully working — go to configuration |

**State C** (current system state) is the most common "first run" scenario and the most important to handle well. The root cause on this system is missing `nvidia-drm.modeset=1`.

## 7. Key Fixes Implementation Detail

### Fix: `nvidia-drm.modeset=1`

**Why needed:** Without this parameter, the NVIDIA kernel DRM module doesn't take full ownership of modesetting, causing USB-C DisplayPort connections to fail on Optimus laptops.

**Current system state:** Parameter is absent from `/proc/cmdline` and not set in `/etc/default/grub`. This is confirmed as the primary reason the plugged-in display shows "disconnected" in xrandr.

### Fix: Force xrandr re-detection (no reboot)

Try this before requiring a reboot — sometimes works immediately:

```bash
xrandr --setprovideroutputsource NVIDIA-G0 modesetting 2>/dev/null
xrandr --auto
```

### Fix: Load appledisplay module

```bash
sudo modprobe appledisplay
```

Module is confirmed available at `/lib/modules/6.8.0-106-generic/kernel/drivers/usb/misc/appledisplay.ko.zst`. Once loaded and USB cable is connected, brightness appears under `/sys/class/backlight/apple_backlight/`.

### Fix: udev rule for brightness

```
SUBSYSTEMS=="usb", ATTRS{product}=="Apple LED Cinema Display", GROUP="users", MODE="0664"
```

Written to `/etc/udev/rules.d/10-apple-cinema-display.rules`.

### Fix: Suspend/resume hook

```ini
# /etc/systemd/system/apple-cinema-resume.service
[Unit]
Description=Restore Apple Cinema Display after suspend
After=suspend.target hibernate.target hybrid-sleep.target

[Service]
Type=oneshot
User=%I
Environment=DISPLAY=:0
Environment=XAUTHORITY=/run/user/1000/gdm/Xauthority
ExecStartPre=/bin/sleep 2
ExecStart=/usr/bin/xrandr --auto

[Install]
WantedBy=suspend.target hibernate.target
```

## 8. Persistence Strategy

**Display layout** → written to `~/.xprofile` as an `xrandr` command block.

```bash
# Apple Cinema Display layout — managed by cinema-display-setup
xrandr --output eDP-1 --primary --mode 1920x1200 --pos 0x0 \
       --output DP-1 --mode 1920x1200 --pos 1920x0
```

**Tool config** → `~/.config/cinema-display-setup/config.json`:
```json
{
  "displays": [
    {"usb_id": "05ac:921d", "output": "DP-1", "layout": "right"}
  ],
  "brightness": 72,
  "suspend_hook_installed": true,
  "udev_rule_installed": true
}
```

**System fixes** (GRUB, udev, systemd) are written to their canonical locations and survive reboots natively.

## 9. Error Handling & Edge Cases

| Scenario | Handling |
|---|---|
| Not running as root when sudo needed | Prompt clearly: "This step requires sudo. Reason: X" |
| `xrandr` output name changes (DP-1 vs DP-2 etc.) | Detect by resolution (1920×1200) + match to USB presence |
| Multiple Cinema Displays connected | Detect all, configure each independently |
| Display connected but unrecognized resolution | Show available modes from xrandr, let user pick |
| GRUB file has non-standard format | Safe regex-based edit; backup original first |
| Update-grub fails | Show error, tell user to run manually |
| Missing `rich` library | Fall back to plain-text menus with ANSI color codes |
| Wayland session (not X11) | Detect and warn; suggest switching to X11 session or note wlr-randr alternative |

## 13. Known Constraints

- **Sudo required** for: GRUB edit, update-grub, modprobe, udev rules, systemd services
- **Reboot required** after: GRUB changes (nvidia-drm.modeset=1) — tool must communicate this clearly
- **X11 only** for xrandr-based layout management; Wayland requires `wlr-randr` or `gnome-randr` — out of scope for v1
- **Brightness via sysfs** only works once `appledisplay` module is loaded AND USB cable is connected to laptop
- **iSight webcam** requires user to source the Apple firmware file — tool can guide but cannot automate the download
